> A product management case study focused on redesigning LinkedIn's Feed and Job Search experience — reducing noise, increasing relevance, and helping job seekers track applications without leaving the platform.

---

## 🎯 Product Goal

Redesign two core LinkedIn experiences for job seekers and professionals:
1. **Feed** — reduce irrelevant content and give users control over what they see
2. **Job Search & Tracking** — simplify navigation and build an application pipeline inside LinkedIn

**Target users:** Job seekers (students, recent grads, career changers) and professionals using LinkedIn for industry news.

---

## 🔍 Problem Statement

### Problem 1 — Chaotic and irrelevant feed
- Users see promotional posts and self-congratulatory updates instead of industry content
- No way to filter feed by topic or content type
- Algorithm rewards engagement bait over quality content
- **Result:** Users open LinkedIn less often and trust the platform less

**Hypothesis:** If users can filter and customize their feed by content type, session quality and daily active use will increase.

### Problem 2 — Broken job search and no application tracking
- Too many clicks from job discovery to application
- No built-in tracker — users rely on Notion or spreadsheets externally
- Filters are hidden and hard to combine (location + salary + seniority)
- **Result:** Job seekers feel frustrated and abandon mid-search

**Hypothesis:** If users can track application status in one place, they will apply more confidently and return to LinkedIn more often.

## 📖 User Stories

### Feed (5 stories)

| # | As a... | I want to... | So that... |
|---|---|---|---|
| 1 | Daily LinkedIn user | Filter feed by content type (jobs, articles, updates) | I only see content relevant to my professional goals |
| 2 | Professional following many connections | Hide promotional posts with one click | My feed shows real insights, not marketing noise |
| 3 | Job seeker using LinkedIn daily | See a separate Jobs Feed tab | I don't mix personal browsing with job hunting |
| 4 | Content creator | See which posts reached relevant audiences | I can improve my content strategy |
| 5 | New LinkedIn user | Set up feed interests during onboarding | My feed is relevant from day one |

### Job Search & Tracking (5 stories)

| # | As a... | I want to... | So that... |
|---|---|---|---|
| 6 | Active job seeker | See all applications in one dashboard with status | I never lose track of opportunities |
| 7 | Job seeker filtering positions | Combine all filters in one panel | I find the right jobs faster |
| 8 | Applicant who applied last week | Get notified when recruiter views my profile | I know my application is being considered |
| 9 | Job seeker comparing offers | Save listings to a Compare board | I make informed decisions without 10 browser tabs |
| 10 | Junior looking for first job | See an Entry-level only quick filter | I stop wasting time on senior roles |

## ⚡ MoSCoW Prioritization

| Priority | Feature | Reasoning |
|---|---|---|
| 🔴 Must Have | Application Status Dashboard | Core problem — users track apps externally |
| 🔴 Must Have | Feed Content Type Filter | Direct fix for Problem 1. High impact, simple toggle |
| 🔴 Must Have | Hide Post — "Not interested" | Minimal effort, immediate UX improvement |
| 🟠 Should Have | Combined Job Filters (one panel) | Reduces friction significantly |
| 🟠 Should Have | Entry-level only quick filter | Huge value for target user — students and juniors |
| 🟠 Should Have | Recruiter profile view notification | Adds transparency and trust |
| 🟡 Could Have | Jobs Feed — separate tab | Nice, but requires behavior change |
| 🟡 Could Have | Job Comparison Board | Power user feature — good for retention |
| ⚪ Won't Have | AI-generated weekly digest | Out of scope for this case study |
| ⚪ Won't Have | Recruiter-facing dashboard | Different user persona — out of scope |

## 🔄 Sprint Retrospective

**What went well:**
- Problem definition was strong — real personal pain points made the case study convincing
- MoSCoW helped avoid scope creep early
- All 3 wireframes delivered with PM annotations

**What didn't go well:**
- No real user interviews — assumptions based on personal experience only
- Success metrics not fully defined per feature

**Key PM insight:**
> The strongest part of this case study is the problem definition. The weakest is lack of user validation. In a real role, you would run at least 3 user interviews before writing stories. This is the next skill to develop.

## 🧠 What I Learned as PM

- How to frame a product problem using **How Might We** and hypothesis format
- How to write **user stories** that focus on value, not features
- How to use **MoSCoW** to make scope decisions under time constraints
- How **wireframes serve PM thinking** — annotations matter more than visual polish
- Why **scope control** (saying Won't Have) is as important as building features
